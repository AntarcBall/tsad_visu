Strategy split

먼저 결정해야 할 두 개의 축

논문은 detection 목적과 update mechanism이 달라지면 같은 모델 계열도 완전히 다른 장단점을 가진다고 봅니다.

Real-time detection

실제 anomaly가 발생하는 즉시 잡아야 하는 운영입니다. latency budget, alert fatigue, compute envelope이 중요하며, GRU/CNN 기반 reconstruction workflow나 경량 prediction scheme이 실용적일 수 있습니다.

  • 맞는 상황: CPS, 제조 설비 모니터링, online business monitoring
  • 핵심 질문: 초당 몇 번 scoring 가능한가?
  • 주의점: false positive가 운영팀 신뢰를 빠르게 무너뜨릴 수 있음

Early warning

실제 breakdown 전에 precursor signal을 감지하는 운영입니다. prediction error, trend deviation, maintenance lead time이 핵심이며 LSTM/HTM류 접근이 잘 맞을 수 있습니다.

  • 맞는 상황: predictive maintenance, scheduled intervention, spare-part planning
  • 핵심 질문: 얼마나 일찍 경고해야 실제 action으로 이어지는가?
  • 주의점: 너무 이른 경고는 precision 하락과 해석 부담을 유발

Sliding window

window size가 곧 문제 정의입니다. window가 짧으면 precursor를 놓치고, 길면 계산량과 label alignment 난도가 커집니다.

  • 장점: CNN/TCN/Transformer 실험과 잘 맞음
  • 리스크: concept drift가 심하면 고정 window가 빠르게 낡음
  • 실무 팁: domain 주기와 sensor sampling rate를 같이 보세요

Incremental update

streaming 환경에서는 새 데이터가 모델을 계속 밀어냅니다. adaptive threshold와 forgetting control이 중요합니다.

  • 장점: distribution shift 대응이 빠름
  • 리스크: anomaly를 정상으로 학습해 버리는 contamination 위험
  • 실무 팁: update trigger와 rollback rule을 명시하세요

Threshold policy

좋은 anomaly score도 threshold가 나쁘면 실패합니다. 논문 메시지는 모델만큼 threshold strategy가 중요하다는 점입니다.

  • static: 단순하지만 drift에 약함
  • adaptive: 환경 변화에 강하지만 tuning이 필요
  • human-in-the-loop: 운영 신뢰도를 높이지만 비용이 듦
Training & preprocessing

Loss, update, denoising를 따로 보지 말고 연결해서 보기

Loss function

어떤 residual을 더 크게 벌점 주는지에 따라 anomaly score의 민감도가 달라집니다.

  • reconstruction loss: 정상 패턴 복원 실패를 직접 포착
  • prediction loss: future deviation을 통해 early warning과 연결
  • adversarial / hybrid loss: 표현력을 높이지만 안정성 관리 필요

Batch learning vs online update

batch learning은 재현성과 관리가 쉽고, online update는 drift 대응력이 좋습니다. 정답은 하나가 아니라 운영 리듬에 따라 달라집니다.

  • batch: 검증과 롤백이 쉬움
  • online: 빠르지만 catastrophic forgetting 위험
  • semi-online: 주기적 refresh로 절충 가능

Denoising

denoising은 false alarm을 줄이는 데 유용하지만, 약한 precursor signal까지 지우면 early warning 성능이 무너질 수 있습니다.

  • 언제 유리한가: 센서 스파이크와 measurement noise가 큰 경우
  • 언제 위험한가: subtle precursor가 중요한 고장 예측
  • 체크포인트: raw와 denoised score를 함께 비교
Scenario playbook

논문 메시지를 실제 선택 시나리오로 번역

아래 카드는 논문 본문의 guideline map을 실무 판단 문장으로 바꾼 것입니다.
Scenario A

고차원 · relation-heavy · 즉시 대응

예: WADI류 plant monitoring, 복수 센서/actuator 연동

  • 우선순위: relation-aware modeling + low-latency scoring
  • 추천 관점: graph-aware feature relation, compact threshold loop
  • 주의점: 큰 모델보다 operator explainability가 더 중요할 수 있음
Scenario B

중간 차원 · breakdown 조기 경고

예: predictive maintenance, failure precursor 탐지

  • 우선순위: prediction error, warning lead time, missed anomaly 최소화
  • 추천 관점: LSTM/GRU 계열 + adaptive threshold
  • 주의점: 너무 잦은 경고는 현업이 무시하게 만듦
Scenario C

label scarce · baseline이 먼저 필요한 경우

예: 새로운 산업 현장, annotation 거의 없음

  • 우선순위: unsupervised reconstruction baseline 확보
  • 추천 관점: AE/VAE류로 시작하고 benchmark 해석을 곁들임
  • 주의점: baseline의 F1만 보고 운영 투입하지 말 것
Scenario D

drift 심함 · streaming update 필수

예: cloud telemetry, nonstationary workload

  • 우선순위: incremental adaptation, contamination guardrail
  • 추천 관점: update cadence + rollback + alert review loop
  • 주의점: anomaly segment를 정상 패턴으로 흡수하지 않도록 검증 필요

의사결정 체크리스트

  • 탐지 목적이 즉시 반응인지 조기 경고인지 분명한가?
  • window size / update cadence가 데이터 sampling rate와 맞는가?
  • threshold를 누가 관리하고, drift 시 누가 조정하는가?
  • denoising이 실제 precursor signal을 지우지 않는가?
  • 정확도 외에 latency, operator burden, rollback cost를 측정하는가?
